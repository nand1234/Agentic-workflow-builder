## Auth & Session Setup

How tests establish an authenticated session:

```js
// Standard "guest beneficiary" flow used by 90% of E2E booking tests.
// Generates a fresh voucher via VMS, gets an SSO token via booking, then
// visits the start-booking redirect URL (which lands on Landing/MapSplitView).
cy.registerVoucherwithoutShireToBooking("stay");           // boxType: stay | nonStay | ownuse | crossell | …
cy.registerVoucherwithoutShireToBooking("stay", 0, 0);     // optional: grossMarginExpectationPercent, exchangeValue

// Same flow, but caches the voucher across its() blocks within a describe so
// subsequent tests reuse the same voucher (faster integration suites).
cy.setsessionWithoutShireToBooking("stay");

// Logged-in beneficiary variant — registers the voucher to a real Shire account first.
cy.registerVoucherAndLoginShireToBooking("stay");
cy.registerVoucherWithCorrectGoldenIDInShire("stay", 0);

// Own-use journey — fetches a login_token from the shire DB via cy.task("queryDb"),
// then visits /login/authenticate-token/:token/start-own-use?…
cy.startOwnUse({ experienceId, stayName, searchStartDate, searchEndDate, brand, country });

// API-only smoke tests skip the UI session entirely and just generate a voucher:
cy.voucherForPerformBooking("stay");
cy.get("@apivoucherid").then(v => cy.PerformStayBookingRequest(v, …));
```

**Notes:**
- All session helpers make real HTTP calls against the configured backend (preprod / devint / review / local). Nothing is stubbed at the auth layer.
- `cy.registerVoucherwithoutShireToBooking` and `cy.setsessionWithoutShireToBooking` differ only in caching: the `setsession*` variant keeps `previousVoucher`/`previousBoxType` in module state and reuses the same voucher when the boxType matches.
- The session is never cached via Cypress' built-in `cy.session()`. Cookies and localStorage are wiped before every test (see global `beforeEach` below).
- Voucher IDs are exposed to the test via the alias `@voucherID` (UI flows) or `@apivoucherid` (API flows).

**Global beforeEach (from `cypress/support/e2e.js`):**
```js
beforeEach(() => {
    cy.clearAllCookies();
    cy.clearAllLocalStorage();
    global.emailAddress = `booking${Math.floor(Math.random() * 999999999 + 1)}@smartbox.com`;
    cy.intercept("https://book-pp.smartbox.com/**", req => {
        req.continue(res => {
            if (res.statusCode >= 400) {
                failedRequests.push({ url: req.url, method: req.method, status: res.statusCode });
            }
        });
    });
});

// Plus a global afterEach that pretty-prints a table of any 4xx/5xx XHRs to the terminal
// via cy.task("logToTerminal", …) and resets the buffer.
//
// Plus a global before() that stamps `global.executionStartTime`.
//
// Plus a global Cypress.on("uncaught:exception", …) that suppresses one specific
// Worldline payment-page error: "Cannot read properties of null (reading 'addEventListener')".
//
// Most individual specs ALSO declare a local beforeEach that swallows ALL
// uncaught exceptions:
//     beforeEach(() => cy.on("uncaught:exception", () => false));
// This is the de-facto standard at the spec level.
```

## Navigation

How tests navigate to pages:

```js
// Custom navigation commands (defined in cypress/support/commands.js) are
// preferred — every test starts from one of these:

cy.navigateTolandingPage();                                          // /:lang/landing
cy.navigateToSearchPage();                                           // /:lang/search
cy.navigateToUniversalDetailPage(boxId, experienceId, detailType);   // detailType ∈ "experience" | "noExperience" | (anything else → /detail/:boxId/:boxId:experienceId)
cy.navigateToUniversalConfirmation(bookingRef);                      // /:lang/confirmation-details/:ref (or trailing nothing)
cy.upcomingBooking(bookingRef);                                      // /:lang/upcoming-details/:ref

// Auth helpers above (registerVoucher* / setsessionWithoutShireToBooking / startOwnUse)
// also visit() under the hood — they ARE the entry-points to the booking SPA.

// Page Objects expose verifyUrl()-style guards rather than raw cy.url() —
// e.g. LandingPage.verifyUrl(), MapSplitViewPage.verifyUrl(),
// UniversalCheckoutPage.verifyCheckoutPageURL(), ConfirmationPage.verifyUrl().

// Raw cy.visit() is used only inside support helpers (commands.js,
// LoginShirePage.js); spec files do not call cy.visit() directly.
```

**Available navigation commands:**

| Command | Arguments | Navigates to |
|---|---|---|
| `cy.navigateTolandingPage` | `()` | `${jarvisBaseUrl}/:lang/landing` |
| `cy.navigateToSearchPage` | `()` | `${jarvisBaseUrl}/:lang/search` |
| `cy.navigateToUniversalDetailPage` | `(boxID, experienceId, detailType)` — `detailType ∈ "experience" \| "noExperience" \| any` | `${jarvisBaseUrl}/:lang/detail/:boxID/:experienceId[?is_experience=true]` |

## Selector Style

The selector pattern used consistently across the suite:

```js
// Canonical: raw attribute selector via cy.get(), with the locator
// stored as a string constant in an ELEMENT object at the top of each
// page-object class.
const ELEMENT = {
    firstName:    "[data-testid=first-name-input]",
    SummaryCard:  "[data-testid=universal-checkout-summary-card]",
    SummaryCardPersonNightsInfo: "[data-testid=person-nights-info-updated], [data-testid=person-nights-info]", // multi-fallback
};
cy.get(ELEMENT.firstName).should("be.visible").type("Jane");

// No cy.getByTestId() / cy.findByTestId() helpers exist — Testing Library
// is NOT installed.
```

**The `data-testid` attribute name used:** `data-testid` (canonical, 426 occurrences in support code)

**Inconsistencies found:**
- `cypress/support/gui/booking/base/SearchFilters.js` uses `data-test-id` (with a hyphen) for the trip-advisor facet — 6 selectors total: `tripAdvOptions`, `tripAdvOptionsMobile`, `tripAdvRadio`, `tripAdvRadioMobile`, `tripAdvDescription`, plus its mobile-overlay scope. This mirrors the DOM attribute that the underlying search component renders. Treat as legacy outlier and use `data-testid` for everything else.
- A handful of selectors compose `[data-testid=…]` with CSS-class scopes (e.g. `".form-action-button [data-testid=form-action-button]"`, `".checkout-bottom-banner [data-testid=form-action-button]"`) to disambiguate the same testid rendered in two places (desktop vs mobile slot). This is acceptable when the testid alone is not unique.
- `[id=terms-and-conditions-checkbox-error]` and `[id$='-error']` (form errors) are used in `UniversalCheckoutPage.js` because error messages are addressed by id, not data-testid.

## Assertion Style

```js

    assert.equal(response.status, 204);
    assert.containsAllKeys(response.body, ["code", "message", "error", "transaction_code"]);

});
expect(response.body).to.have.property("success", true);
expect(response.body).to.have.property("uuid").and.to.be.a("string");
expect(detailPagePrice).to.eq(checkoutPrice);

// No custom Chai matchers, no cypress-image-snapshot, no Testing Library

// Cypress.expose("enablePerformanceValidation") === true and currently disabled
// in every npm script.
```

## Viewport & Mobile Handling

```js
// Desktop and mobile are SEPARATE Cypress configs, not separate context blocks.
// One spec is shared by both — viewport is set globally by the chosen
// config-file at run time:
//
//   cypress run --config-file cypress-desktop.config.js   →  viewport 1280x720
//   cypress run --config-file cypress-mobile.config.js    →  viewport  390x812
//
// Inside the spec, branching is done by reading Cypress.config("viewportWidth"):
if (Cypress.config("viewportWidth") == 1280) {
    UniversalCheckoutPage.clickFormActionButton();           // desktop path
    ConfirmationPage.verifyUrl();
} else {
    UniversalCheckoutPage.clickOnReviewSummaryButton();      // mobile path
    UniversalCheckoutPage.clickFormActionButton();
}
//
// Page objects often expose a `getViewPort(1280)` helper (on CommonPage)
// that returns truthy for desktop and is used to pick the right selector
// (e.g. desktopBackButton vs mobileBackButton).
//
// cy.viewport() is NEVER called from spec code, and no `context('Mobile', …)`
// blocks exist. Mobile-only / desktop-only specs are filtered with
// @cypress/grep tags (e.g. @sbx_fr, @smoke_fr2) plus the right config-file.
```

**Mobile viewport used:** `390 × 812` (iPhone-X-sized, set in `cypress-mobile.config.js`)
**Desktop viewport used:** `1280 × 720` (set in `cypress-desktop.config.js`)

## Running Tests (Environments & Invocation)

**Always run from `cypress/` as the working directory** — the npm scripts and config files assume this cwd.

### Environments (keys in `cypress/support/settings/booking-domains.json`)

| Env key | Backend it points at | Notes |
|---|---|---|
| `local` | `*.defuse.sbxtest.net` (your laptop's docker stack) | Combine with `backend=preprod` or `backend=devint` to target a real backend from a local UI |
| `ie1-devint` | `*-devint.sbxtest.net` | Shared dev-integration env |
| `ie1-preprod` | `book-pp.smartbox.com`, `book-pp.bongo.nl`, etc. | Stable pre-production — preferred for stable smoke runs |
| `ie1-review` | `booking-review-pp-CI_COMMIT_REF_SLUG.review.sbxtest.net` | **Per-branch ephemeral deployment.** Requires `reviewdomain=<slug>` to substitute `CI_COMMIT_REF_SLUG` in the URL template |

### The `--expose` flag (project convention — not standard Cypress)

The suite reads runtime parameters via `Cypress.expose("<key>")` (used in `support/lib/ConfigHelper.js`, `support/commands.js`, `support/services/*.js`). They are passed on the CLI as a single comma-separated `--expose` argument:

```bash
cypress run \
  --expose environment=ie1-preprod,brand=sbx,country=fr,language=fr,grepTags=@sbx_fr,enablePerformanceValidation=false \
  --config-file cypress-desktop.config.js \
  --spec ./tests/E2E/Jarvis/<file>.test.js
```

A handful of older scripts (`cypress:local:validate:e2e`, `docker:local:validate:e2e`) use `--env` instead — treat that as legacy. **All `cypress/package.json` open/run scripts written since use `--expose`.** When adding a new script, copy the closest existing one rather than mixing flags.

### Standard parameter set

| Key | Required | Example values |
|---|---|---|
| `environment` | yes | `local`, `ie1-devint`, `ie1-preprod`, `ie1-review` |
| `reviewdomain` | only when `environment=ie1-review` | branch slug, e.g. `bn-5740-v1` (lowercased GitLab `CI_COMMIT_REF_SLUG`) |
| `backend` | only when `environment=local` | `preprod`, `devint` |
| `brand` | yes | `sbx`, `bon`, `dak`, `emo`, `lav`, `cbx` |
| `country` | yes | `fr`, `it`, `be`, `nl`, `es` |
| `language` | yes | `fr`, `it`, `nl`, `es`, `en` (matches store config) |
| `grepTags` | yes | `@sbx_fr`, `@sbx_it`, `@bon_be`, `@smoke_fr1`, `@preprod`, `@spreprod` (comma-separated for multi-tag filter) |
| `enablePerformanceValidation` | optional | `false` (default in every shipped script — Lighthouse is gated off) |

`brand`/`country`/`language` must be a valid tuple matching `booking-domains.json` (e.g. `bon` + `be` + `nl` is valid; `bon` + `fr` + `fr` is not).

### How to run on **review**

Review deployments are per-branch ephemeral. The `apiBaseUrl` template is
`https://booking-review-pp-CI_COMMIT_REF_SLUG.review.sbxtest.net` and `Cypress.expose("reviewdomain")` substitutes the `<slug>` part.

```bash
# Headless desktop run on a review env
cypress run \
  --expose environment=ie1-review,reviewdomain=<BRANCH-SLUG>,brand=sbx,country=fr,language=fr,grepTags=@sbx_fr,enablePerformanceValidation=false \
  --config-file cypress-desktop.config.js \
  --spec ./tests/E2E/Jarvis/<file>.test.js

# Interactive (canned script — only the desktop:fr variant exists today):
bun run cypress:open:desktop:review:fr
# ↑ note: the slug is hard-coded in cypress/package.json:22 — edit it before running
```

**Finding the review slug for a branch:**
- GitLab uses `CI_COMMIT_REF_SLUG` — the branch name lowercased with non-alphanumerics replaced by `-`. Branch `BN-5740-V1` → slug `bn-5740-v1`.
- The actual deployed slug is visible in the GitLab MR pipeline page (the "Review App" button) — review deployments may be expired or never built; if `https://booking-review-pp-<slug>.review.sbxtest.net` returns 404, the deployment isn't live and you must re-trigger the pipeline or use `ie1-preprod` instead.
- A review env is **only useful when the branch contains backend or template changes** that need to round-trip through a deployed booking instance. For pure Vue-frontend changes, prefer `local` with `backend=preprod` instead — it's faster and doesn't require a working pipeline.

### Adding a new permanent run script

When a new test (or new brand/country/env tuple) needs a recurring run command, add a script to `cypress/package.json` next to the closest existing one. Naming convention is `<verb>:<viewport>:<env>:<store>`:

```
cypress:open:desktop:review:fr
cypress:open:mobile:devint:bn
cypress:local:desktop:preprod:it
docker:local:mobile:devint:be
test:desktop:E2E
```

Always include `enablePerformanceValidation=false` unless you intentionally want to run Lighthouse.

### Common gotchas

- **`baseUrl` is unset** — every URL is built dynamically from `ConfigHelper`. Do not pass `--config baseUrl=...` on the CLI; it has no effect.
- **`allowCypressEnv: false`** is set in both config files. This blocks the standard `CYPRESS_*` env-var → `Cypress.env(...)` shortcut. Use `--expose` only.
- **`cypress run` from inside `cypress/`** — every `--spec` path is relative to that cwd. Paths starting with `cypress/...` (i.e. doubled-up) won't match.
- **Spec filtering — `--spec` vs `grepTags`** — `--spec` filters files; `grepTags` filters describe/it blocks within those files via `@cypress/grep`. Use `--spec` to narrow to one file fast; use `grepTags` to slice across many files (`@smoke_fr1` etc.).
- **`--browser chrome`** is hard-coded only in the `*:validate:e2e` chains. Other scripts let Cypress pick (Electron by default). Add `-b chrome` if a flow only works in real Chrome (e.g. ApplePay/GooglePay redirects).
- **Cookies/local-storage are wiped on every test** — there is no `cy.session()` reuse. Each spec re-registers a fresh voucher. Long suites are slow on preprod for this reason.

## Describe / It Naming Convention

```
// Sentence case, narrative, plain English. Mix of three styles is in active use:
//
// 1) "Test {feature}" / "Check {feature}" / "{feature}" — feature-area headers
describe("Universal Checkout", { tags: ["@sbx_it"] }, () => { … })
describe("Test Landing Page functionality", { tags: ["@sbx_fr", "@set1"] }, () => { … })
describe("Instant Booking and Cancellation on the universal checkout", { tags: ["@sbx_it"] }, () => { … })
describe("check stay booking booking using API /api/rest/jarvis/v1/performBooking", { tags: ["@sbx_fr", "@set1", "@smoke_fr1"] }, () => { … })
describe("OPP Booking Journeys", { tags: ["@sbx_fr", "@set2"] }, () => { … })

// 2) "should {behaviour}" — most common for it() blocks
it("should book an inbox stay and cancel it on the universal checkout", () => { … })
it("should display the sections for inbox stay experiences", () => { … })

// 3) "As a beneficiary I want to {behaviour}" — BDD persona style for user-facing E2E
it("As a beneficiary I want to pay with mastercard on the universal checkout", () => { … })
it("As a beneficiary I want to book non stay experience", { tags: ["@smoke_fr2"] }, () => { … })

// EVERY describe and most it() blocks declare a tags option — these drive
// @cypress/grep filtering. Common tag families:
//   Brand/store:     @sbx_fr, @sbx_it, @bon_be, @smoke_fr1, @smoke_fr2, @smoke_it
//   Suite slice:     @set1, @set2, @set3, @smoke, @preprod, @spreprod
//
// describe.skip(...) is used to disable disabled-feature suites
// (e.g. cross-sell — see cross.sell.booking.test.js / CrossSellConfirmation.test.js).
//
// Casing: descriptions are inconsistent (some lowercase, some Title Case);
// tense is mixed ("should", "Test", "Check", "As a beneficiary"). No trailing
// period. No nested describes — every spec has top-level describe blocks only.
```

## Wait / Timing Patterns

```js
// PRIMARY pattern: intercept-and-wait via @-aliases. Aliases are registered
// inside dedicated interceptor classes under
// cypress/support/gui/booking/interceptors/{Availability,Booking,Friday,Search,Voucher}ApiInterceptor.js
// and consumed in the spec with cy.wait("@alias").
BookingApiInterceptor.interceptCheckoutDetailRequest();      // sets up cy.intercept(...).as("checkoutDetailRequest")
```

## Known Inconsistencies

- **`data-test-id` vs `data-testid`** — `cypress/support/gui/booking/base/SearchFilters.js` uses `data-test-id` (with hyphen) for 6 trip-advisor facet selectors because the underlying search component renders that DOM attribute. All other support code (~426 occurrences) uses `data-testid`. Treat `data-testid` as canonical for new tests; only use `data-test-id` when scraping confirms it on the live element.
- **Bare `cy.wait(ms)` smells** — 12 occurrences across 6 files (`ownuse.test.js`, `UniversalCheckout.test.js` ×3, `OnlineCategories.test.js` ×2, `UniversalDetailPage.test.js`, `MSVListAndMapMSR.test.js` ×5). Replace with intercept-aliases or element-based waits in any new test. Bare waits inside the `checkBookingcomplete*` retry helpers are intentional polling — leave them alone.
- **`should("be.exist")` typo** — `UniversalCheckoutPage.clickOnBackButton` and a few page-objects use `.should("be.exist")` which is not a valid Chai chainer (correct form is `.should("exist")`). Cypress passes it because the assertion gets coerced, but it should be `"exist"` in new code.
- **Two cross-sell suites disabled via `describe.skip`** — `cross.sell.booking.test.js` and `CrossSellConfirmation.test.js` are intentionally skipped per BN-5574. Do not re-enable when adding new cross-sell tests until that ticket is resolved.
- **Mixed `it()` naming** — three styles coexist (`should …`, `Test …`, `As a beneficiary I want to …`). Match the style already in the file you are editing rather than introducing a fourth. New E2E user-journey tests should use `As a beneficiary I want to …`; new integration / API tests should use `should …`.
- **No `cy.fixture()` usage** — fixtures are JS modules imported with ES `import`, never JSON loaded via `cy.fixture()`. Don't introduce JSON fixtures unless there's a strong reason.
- **`baseUrl` is unset** — every navigation goes through `ConfigHelper.get*BaseUrl()` and the runtime `--env environment=…,brand=…,country=…,language=…` flags. Do not add `cy.visit("/some/path")` with a relative path; either use the existing `cy.navigateTo*` commands or build the URL via `ConfigHelper`.
- **Voucher state across tests** — `cy.setsessionWithoutShireToBooking` reuses the previous voucher when the boxType matches. If a test mutates the voucher (cancellation, OOB), use `cy.registerVoucherwithoutShireToBooking` instead so the next test gets a fresh one.
- **Spec layout** — pattern is `cypress/tests/{E2E,integration}/**/*.test.js`, not `cypress/e2e/**/*.cy.{js,ts}`. The selector-patterns reference shows `cy.getByTestId()` as a possible helper; this project does not define one — always use `cy.get('[data-testid=…]')`.
